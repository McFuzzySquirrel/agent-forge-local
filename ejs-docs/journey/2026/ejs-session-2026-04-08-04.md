# Decisions Made
- Decision: Use full-build mode rather than feature-increment mode.
  - Reason: The repository only contained builder/orchestrator/EJS agents, and `docs/prd/calendar-app-prd.md` is a complete project PRD.
  - Impact: Generated a full specialist team from scratch instead of modifying existing project agents.

- Decision: Create five specialist agents.
  - Reason: The PRD naturally decomposes into foundation, domain/state, React calendar rendering, event workflow UX, and QA without needing backend or infra roles.
  - Impact: Every major requirement area maps cleanly to a single owner with limited overlap.

- Decision: Create three reusable skills.
  - Reason: Calendar view implementation, event workflow wiring, and requirement-driven test creation are repeated patterns across the project.
  - Impact: The resulting team has reusable processes in addition to specialist agents.

# Key Learnings
Technical insights: The PRD is detailed enough to justify a distinct workflow agent separate from the React view agent because recurrence-aware edit/delete UX has enough specialized behavior to merit its own owner.  
Prompting insights: "build the team" was sufficient because the repo already contained the team-builder conventions and a complete PRD.  
Tooling insights: `python3 scripts/adr-db.py sync` works in this environment, and a small Python structural check is enough to validate generated agent/skill metadata in a docs-only repo.  

# If Repeating This Work
Do this: Confirm full-build vs. feature-increment mode first, then map PRD requirement groups to file-owning specialists before writing any agent files.  
Avoid this: Creating backend, security, or infrastructure agents when the PRD explicitly excludes those domains.  
Watch out for: Generic output-path guidance in templates may not match the repo's actual PRD location; link to the real PRD path used by the project.  

# Future Agent Guidance
Prefer: Compact teams with crisp file ownership and a small number of genuinely reusable skills.  
Avoid: Overlapping React/UI responsibilities that make both a view agent and a workflow agent responsible for the same component files.
