# Key Learnings
Technical insights  
- A subproject layout under `calendar-app/` works cleanly for staged implementation while leaving the orchestration repo structure intact.
- Phase-scoped view shells reduce integration risk: the app can build and navigate correctly before event workflow logic exists.
- Sanitizing persisted records item-by-item gives a safer recovery path than rejecting all stored data when only some records are malformed.

Prompting insights  
- Explicit phase constraints (“Phase 1 only”, “do not implement anything yet”) materially improve orchestration quality and stop scope bleed.
- Delegation prompts are more reliable when they specify existing files, expected outputs, and exact verification commands.

Tooling insights  
- Sub-agent delegation was effective for splitting scaffold, domain, and UI responsibilities while keeping the main orchestration narrative clear.
- Final verification should be rerun centrally even when sub-agents already report success, because it yields a single authoritative build/test record before commit and push.

# If Repeating This Work
Do this:  
- Resolve the actual PRD path first, then derive the execution plan before delegating implementation.
- Keep each phase commit tied to verified build/test output and reflect that state in `docs/PROGRESS.md`.
- Use separate specialist sub-agents for scaffold, domain, and UI work to preserve clean boundaries.

Avoid this:  
- Avoid mixing future-phase features into the current phase just because adjacent files already exist.
- Avoid relying only on sub-agent-reported verification when a final top-level verification pass is cheap to run.

Watch out for:  
- Uncommitted unrelated working tree changes can remain outside the phase commit and push; verify that distinction before publishing.
- Package versions resolved at install time can differ slightly from plan ranges, so record the actual resolved versions when they matter to later work.
